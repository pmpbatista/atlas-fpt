# Spec: Local backup — manual + scheduled (#9)

## Problem

The Room DB holds the user's full financial life. Uninstalling the app, swapping devices, or DB corruption means total loss. CSV import covers only transactions. The user wants a local-only backup story they control.

## Decisions

- **Local-only.** Folder picked via SAF (`ACTION_OPEN_DOCUMENT_TREE`) with persistable URI permission. No cloud, no auth.
- **Format.** Zip archive containing `atlas.db` (+ `-wal`/`-shm` if present) and `manifest.json` (`appVersion`, `dbVersion`, `createdAt`, `schemaHash` placeholder).
- **Filename.** `atlas-backup-YYYYMMDD-HHmmss.zip` (the issue used `spendtrack-…`; using the rebranded prefix per [[atlas-financial-planner-tracker]]).
- **Atomicity.** Before copying the DB, run `PRAGMA wal_checkpoint(TRUNCATE)` to flush the WAL. Write to a `.tmp` SAF child, then rename to the final filename.
- **Schedule.** Periodic WorkManager job at one of three cadences (Daily / Weekly / Monthly) when the user enables it. Off by default.
- **Retention.** Optional "Keep last N" (default 5, toggle to disable). After a successful run, delete older `atlas-backup-*.zip` files in the same folder.
- **Restore.** Out of scope. Manifest captures `dbVersion` to make a future restore decision easier.
- **Encryption.** Out of scope.
- **SharedPreferences/settings inclusion.** Out of scope for v1 — settings are recoverable manually. Manifest reserves a field for them in the future.

## Settings additions

- `backupFolderUri: String?` (the persisted tree URI)
- `backupScheduleEnabled: Boolean` (off by default)
- `backupFrequency: BackupFrequency` (`DAILY` / `WEEKLY` / `MONTHLY`, default `WEEKLY`)
- `backupRetentionCount: Int` (default `5`, `0` = keep all)
- `lastBackupAt: Long?` (millis)
- `lastBackupFilename: String?`

## Components

- **`BackupRepository`** — orchestrates the snapshot:
  1. `db.openHelper.writableDatabase.query("PRAGMA wal_checkpoint(TRUNCATE)")`
  2. Pick the user's folder via the saved URI; create a `.tmp` zip file via SAF.
  3. Stream the DB file (and its WAL/SHM) into the zip.
  4. Append `manifest.json`.
  5. Rename `.tmp` to the final filename.
  6. Update `lastBackupAt` + `lastBackupFilename` in settings.
  7. Apply retention.
- **`BackupWorker`** — Hilt-injected `CoroutineWorker` that calls the repository's `runBackup()`. Registered as periodic unique work `atlas_backup` when scheduling is enabled. Cancelled when disabled.
- **`BackupScheduler`** — small helper to translate `BackupFrequency` → `PeriodicWorkRequest` repeat interval and enqueue/cancel.
- **Settings UI** — new Backup section: folder picker row, "Back up now", schedule toggle + frequency dropdown, retention setting, "Last backup" line.

## Error handling

- Folder permission revoked: zero-out `backupFolderUri`, surface a snackbar prompting re-pick.
- Out of space / IO failure: snackbar with the underlying message; no partial files survive thanks to the `.tmp` write-then-rename.
- WAL checkpoint failure: log + continue (worst case, a slightly less consistent snapshot — better than nothing).

## Acceptance

- A "Backup" section appears in Settings with folder picker, "Back up now", scheduled toggle, frequency, retention.
- "Back up now" produces a single zip in the chosen folder containing the DB and a manifest. Reload of the file lists confirms.
- Enabling the schedule registers periodic `atlas_backup` work; disabling cancels it.
- Last backup row reflects the latest successful run; an explicit "Never" otherwise.
- Permissions revocation is handled — the next attempt prompts the user.
- A unit test covers manifest content (json shape) and a fake-folder backup flow exercising retention.

## Open questions, resolved

- **Periodic interval bounds.** WorkManager periodic minimum is 15 minutes — Daily/Weekly/Monthly all fit comfortably.
- **Where to store the URI.** SharedPreferences alongside other settings. Persisting via `contentResolver.takePersistableUriPermission` so it survives reboot.
