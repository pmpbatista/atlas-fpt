## Steps

1. **Settings.** Extend `AppSettings`. Defaults: `WEEKLY`, retention `5`, off, no URI.
2. **SettingsRepository.** New setters for each field. `setBackupFolderUri(uri: Uri?)` calls `contentResolver.takePersistableUriPermission` and `release` correspondingly.
3. **BackupRepository.runBackup(): Result.** Pure function returning a sealed result. Steps as in the spec. Uses `DocumentsContract` to create child files via the saved tree URI.
4. **BackupScheduler.apply(settings: AppSettings).** If scheduling enabled and a folder URI exists, enqueue (or replace) the periodic work. Otherwise cancel.
5. **BackupWorker.** Hilt worker, single `runBackup()` call. Returns `Result.success()` on any branch (failures already logged + surfaced through `errorMessage` in settings if relevant).
6. **Settings UI.** New `BackupSection` composable with rows + the SAF launcher (`rememberLauncherForActivityResult(ActivityResultContracts.OpenDocumentTree())`).
7. **MainActivity.** Call `backupScheduler.apply(settingsRepository.settings.value)` once at startup.

## Risks

- **SAF rename API.** `DocumentsContract.renameDocument` requires API 21+ (we're already minSdk 26) but takes the *document URI*, not the *tree URI*. Track returned URIs carefully after `createDocument`.
- **WAL checkpoint** — calling on a Room db handle is permitted. Use `db.openHelper.writableDatabase.query("PRAGMA wal_checkpoint(TRUNCATE)", null).use { it.moveToFirst() }`.
- **File lock during copy.** Acceptable because checkpoint flushes WAL, leaving the main `.db` consistent for the moment we read it. We don't write during the copy because `runBackup` is called from a single coroutine — Room itself may still write from elsewhere, but the WAL pattern means the on-disk `.db` reflects a complete snapshot at the moment of checkpoint.
